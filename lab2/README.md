# Система управления записями пациентов

Система автоматизации учета пациентов, реализованная на Python с использованием графического фреймворка PyQt6. Проект построен на строгом соблюдении архитектурного паттерна MVC, принципов ООП и обеспечивает работу с XML-документами согласно требованиям **Варианта 5**.

## Архитектура проекта

Проект разделен на три основных слоя в соответствии с паттерном Model-View-Controller: данные и бизнес-логика (model), графический интерфейс (view) и управление (controller).

---

### Основные сущности

* пациент - класс `Patient`
* менеджер базы данных - `PatientRepository`
* обработчик XML - `PatientHandler`
* главное окно приложения - `MainWindow`
* компоненты таблицы и интерфейса - `BaseTable`, `PaginationUI`
* контроллер приложения - `PaginationController`

### Реализованные поведения
* операция добавления новых пациентов с базовой валидацией
* операция гибкого многокритериального поиска пациентов
* операция удаления записей по заданным фильтрам
* операция пагинации (постраничного вывода) данных
* операция импорта (SAX) и экспорта (DOM) базы данных в формат XML

### Описание классов

**PaginationController**: Центральный узел управления (Слой Controller). Выступает медиатором между пользовательским интерфейсом (`BaseTable`, `PaginationUI`) и хранилищем данных (`PatientRepository`). Управляет состоянием пагинации (текущая страница, лимиты на страницу, смещения) и координирует динамическую отрисовку данных.

#### Слой данных (Model)
* **Patient**: Сущность данных, представляющая пациента. Содержит атрибуты, регламентированные предметной областью: ФИО, адрес прописки, дата рождения, дата приема, ФИО врача, заключение.
* **PatientRepository**: Инкапсулирует логику управления массивом записей в оперативной памяти. Реализует методы фильтрации и удаления (`filter_records`, `delete_records`). Включает внутренний алгоритм умного поиска (`_matches`), обеспечивающий поиск по префиксам (началу слов), независимость от порядка слов в запросе и игнорирование пунктуации.
* **PatientHandler**: Отвечает за десериализацию данных. Реализует событийно-ориентированный потоковый SAX-парсер (`xml.sax.ContentHandler`) для высокоэффективного импорта XML-файлов. Сохранение данных (экспорт) реализовано с помощью DOM-парсера (`xml.dom.minidom`).

#### Слой представления (View)
* **MainWindow**: Главное окно приложения (наследует `QMainWindow`). Агрегирует меню, панели инструментов, главную таблицу и нижнюю панель пагинации. 
* **BaseTable / PaginationUI**: Изолированные графические компоненты. `BaseTable` настраивает столбцы и растягивает их по ширине окна, а `PaginationUI` формирует макет с кнопками навигации и полем `QSpinBox`.
* **AddDialog / SearchDialog / SearchDeleteWidget**: Модальные окна (`QDialog`) и виджеты, предоставляющие формы для взаимодействия с пользователем. `SearchDeleteWidget` инкапсулирует сложную логику Варианта 5, динамически скрывая и отображая поля ввода (текст или календари `QDateEdit`) в зависимости от выбранного в `QComboBox` условия. `SearchDialog` включает собственную независимую систему пагинации для вывода результатов поиска.

---

Пользователь системы может:
* просматривать каталог пациентов с использованием постраничной навигации
* менять количество отображаемых записей на одной странице
* добавлять новые записи, заполняя необходимые поля через удобные виджеты (включая всплывающие календари)
* искать и удалять записи по сложным составным критериям Варианта 5 (например: конкретный врач ИЛИ дата приема; частичное совпадение ФИО или адреса)
* сохранять текущий массив записей в XML-файл или загружать данные из стороннего XML-документа.

### Интерфейс (Сценарии демонстрации)

Система предлагает полноценный графический интерфейс (GUI). Ниже описаны основные сценарии взаимодействия:

**Сценарий 1: Загрузка данных и пагинация**
1. При запуске открывается главное окно с пустой таблицей.
2. В верхнем меню пользователь выбирает `Файл -> Открыть (SAX)`. Через системный диалог выбирается файл (например, `patients_primary.xml`).
3. Таблица мгновенно заполняется первой страницей записей (по умолчанию 10). На нижней панели отображается статус текущей страницы.
4. Пользователь вводит новое значение в поле `Стр:` (QSpinBox). Контроллер автоматически пересчитывает лимиты и обновляет таблицу. Кнопками навигации (`>`, `<`) осуществляется переход между страницами.

**Сценарий 2: Добавление новой записи**
1. В верхнем меню выбирается `Записи -> Добавить` (или используется кнопка на панели инструментов).
2. Появляется модальное окно `AddDialog` с полями ввода. Для дат автоматически подставляются текущая дата (прием) и дата 20-летней давности (рождение).
3. При попытке сохранить пустую форму система блокирует действие и выводит `QMessageBox` с ошибкой валидации.
4. Пользователь корректно заполняет поля и нажимает "Сохранить". Окно закрывается, объект добавляется в репозиторий, а главная таблица обновляется.

**Сценарий 3: Интеллектуальный поиск и удаление (Вариант 5)**
1. В меню выбирается `Записи -> Поиск`. Открывается специализированное окно с выпадающим списком условий и собственной таблицей.
2. Пользователь выбирает условие "Врач или Дата приема". Интерфейс динамически адаптируется: появляется текстовое поле для ФИО врача и скрытый календарь с чекбоксом "Искать по дате".
3. Пользователь вводит фамилию врача и нажимает "Найти". 
4. Контроллер обращается к умному поиску в модели: строка очищается от точек, слова разбиваются, и алгоритм находит все точные совпадения по началу слов, игнорируя порядок ввода. Таблица в окне поиска заполняется найденными пациентами.
5. Для удаления пользователь открывает `Записи -> Удалить`, выбирает аналогичное условие, вводит данные и подтверждает удаление. Появляется информационное окно о количестве удаленных записей, и пациенты исчезают из главной таблицы.
